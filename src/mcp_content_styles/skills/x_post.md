## 格式要求

### 格式化模式 (mode="format")
保持核心信息，调整为 X 推文格式：
- 单条推文控制在 280 字符内（中文约 140 字）
- 内容长时使用 Thread（线程）格式，每条独立成段
- Thread 首条是钩子，最后一条是总结/CTA
- 适当使用 emoji，但不要过多（每条 0-2 个）
- 标签放最后一条或首条末尾

### 重写模式 (mode="rewrite")
将内容重写为 X 技术社区风格：

**核心原则：简洁 > 详尽，观点 > 描述，对话 > 单播**

1. **语气**：直接、自信、有主见。像跟同行聊天，不像写文章。
2. **开场**：一句话抛出核心观点或反直觉发现，让人想往下读。
3. **正文**：每条推文一个点，信息密度高，不废话。
4. **互动**：抛问题、求反馈、分享决策过程，引发讨论。

### 推文格式规范

**单条推文**（适合快速观点/工具推荐）
- 控制在 280 字符内
- 一个核心观点 + 一个支撑论据
- 可带一张截图或代码片段

**Thread 线程**（适合深度分享/踩坑经历）
- 首条（钩子）：核心结论或反直觉发现，让人想展开
- 中间条：每条一个独立的点，可单独理解
- 末条：总结 + 互动引导 + 标签
- Thread 长度：3-10 条，不要太长
- 每条之间用 "🧵" 或序号连接

### 写作风格

**简洁有力**
- 短句为主，长句拆开
- 删掉所有不必要的修饰词
- 能用一句话说清的不要用两句

**有观点**
- 不要面面俱到，敢于下判断
- "X is better than Y for Z" 比 "X and Y both have pros and cons" 好
- 有争议性的观点更容易引发讨论

**技术社区调性**
- 可以用英文术语（LLM, RAG, MCP, agent, prompt）
- 中英混排自然即可
- 代码片段用截图或 ``` 包裹
- 数据说话：star 数、性能数据、使用量

**避免**
- 不要用营销语气（"震惊""必看""强烈推荐"）
- 不要堆砌 hashtag（最多 2-3 个）
- 不要写成博客摘要（那不是推文）
- 不要用 "首先/其次/最后" 这种报告体
- 不要过度使用 emoji

### 互动技巧
- 抛出选择题："你们用 A 还是 B？"
- 分享决策过程："选了 X 而不是 Y，因为..."
- 求反馈："这个方案有没有更好的做法？"
- Building in public："今天把 XX 功能上线了，分享一下数据"

### 标签规范
- 最多 2-3 个标签
- 放在 Thread 最后一条或单条推文末尾
- 不要在每条推文都加标签
- 常用：#buildinpublic #AI #LLM #opensource #devtools

### 配图建议
- 代码截图（深色主题更受欢迎）
- 性能对比图表
- 架构图（简洁的）
- 产品截图 / demo GIF
- GitHub star 增长截图
